# Specification: v0.5.1 - MCP REST API Integration

## Goal

Refactor the MCP integration so that the MCP server communicates with the application exclusively through the existing REST API.

The MCP server must no longer access the database, repositories, services, or SQLite storage directly.

The MCP server becomes a lightweight adapter between AI agents and the REST API.

This ensures that:

* the web frontend,
* REST API,
* and MCP server

all operate on the same task data source.

This change improves architectural separation and makes the MCP workflow easier to demonstrate in the YouTube video.

---

## Motivation

The original implementation allowed the MCP server to access the database directly.

This introduces several issues:

* potential database path inconsistencies,
* duplicated access paths,
* more difficult troubleshooting,
* harder explanation during demonstrations.

The new architecture should be:

```text
Browser
   │
   ▼
Web Frontend
   │
   ▼
REST API
   │
   ▼
SQLite

Cursor Agent
   │
   ▼
MCP Server
   │
   ▼
REST API
   │
   ▼
SQLite
```

The REST API becomes the single source of truth.

## Functional Requirements

### 1. Remove Direct Database Access

Review the MCP implementation.

Remove all direct usage of:

```text
SQLModel
Session
Repository layer
Service layer
SQLite database
```

from MCP-specific code.

Files likely affected:

```text
src/tasks_mcp_server/mcp_server.py
src/tasks_mcp_server/mcp_tools.py
```

The MCP server must not interact with the database directly.

---

## 2. Introduce REST API Client Layer

Create:

```text
src/tasks_mcp_server/mcp_api_client.py
```

Purpose:

Provide a small wrapper around REST API calls.

Suggested implementation:

```python
class TasksApiClient:
    ...
```

The client should encapsulate:

```text
get_next_task()
get_task(task_id)
list_open_tasks()
mark_task_done(task_id)
```

Use standard HTTP requests.

Keep implementation intentionally simple.

---

## 3. API Base URL Configuration

Add configuration:

```text
TASKS_MCP_API_BASE_URL
```

Default:

```text
http://127.0.0.1:8000
```

The MCP server must be usable without additional configuration when the web application is running locally.

---

## 4. MCP Tool Refactoring

Refactor all MCP tools to use the REST API client.

---

### Tool: get_next_task

Instead of:

```text
database → repository → service
```

use:

```http
GET /api/tasks/next/open
```

---

### Tool: get_task

Use:

```http
GET /api/tasks/{id}
```

---

### Tool: list_open_tasks

Use:

```http
GET /api/tasks?status=open
```

---

### Tool: mark_task_done

Use:

```http
POST /api/tasks/{id}/done
```

---

## 5. Error Handling

The MCP server should gracefully handle:

```text
REST API unavailable
Connection timeout
404 responses
Invalid responses
```

Return user-friendly MCP responses.

Do not expose stack traces.

Example:

```json
{
  "success": false,
  "message": "Unable to contact Tasks REST API."
}
```

---

## 6. MCP Startup Validation

When the MCP server starts, it should not require the REST API to be running.

However:

* tools should detect unavailable API,
* return clear error messages,
* avoid crashing the MCP server.
