# Specification: v0.2.0 - REST API

## Goal

Implement REST API support for the `tasks-mcp-server` project.

This version exposes the existing task storage and service layer through FastAPI endpoints. The API will later be used by the simple frontend and MCP tools.

Do not implement frontend or MCP tools in this version.

---

## Project Context

This project is created as a demonstration application for a YouTube video series about:

* Spec-Driven Development
* Semantic Versioning
* Cursor Agents
* Model Context Protocol (MCP)

The application is intended to run only on localhost.

Production-grade concerns such as authentication, authorization, security hardening, scalability, deployment, containerization, migrations, and cloud hosting are intentionally out of scope.

The primary goal is simplicity, readability, and ease of setup for viewers.

## Functional Requirements

## 1. FastAPI Application

Create:

```text
src/tasks_mcp_server/app.py
```

The app must expose a FastAPI instance:

```python
app = FastAPI(...)
```

Requirements:

* app title: `Tasks MCP Server`
* version: `0.2.0`
* create database tables on startup
* include the task API router
* expose a health endpoint:

```http
GET /health
```

Response:

```json
{
  "status": "ok"
}
```

For this demo project, creating database tables on startup is intentional. The goal is to make the app easy to clone and run locally.

---

## 2. Task API Router

Create:

```text
src/tasks_mcp_server/api/tasks.py
```

Base prefix:

```text
/api/tasks
```

Required endpoints:

```http
GET    /api/tasks
POST   /api/tasks
GET    /api/tasks/next/open
GET    /api/tasks/{task_id}
PATCH  /api/tasks/{task_id}
DELETE /api/tasks/{task_id}
POST   /api/tasks/{task_id}/done
```

Important: define `/api/tasks/next/open` before `/{task_id}` to avoid route conflicts.

---

## 3. Endpoint Behavior

### List Tasks

```http
GET /api/tasks
```

Returns all tasks ordered by creation time descending.

Optional query parameter:

```text
status
```

If provided, return only tasks with the given status.

Supported values:

```text
open
in_progress
done
cancelled
```

---

### Create Task

```http
POST /api/tasks
```

Request:

```json
{
  "title": "Create frontend",
  "description": "Add a simple HTML frontend"
}
```

Response status:

```text
201 Created
```

Response body must contain the created task.

---

### Get Next Open Task

```http
GET /api/tasks/next/open
```

Returns the oldest open task.

If no open task exists, return:

```text
404 Not Found
```

Error body:

```json
{
  "detail": "No open task found"
}
```

---

### Get Task

```http
GET /api/tasks/{task_id}
```

If the task exists, return it.

If the task does not exist, return:

```text
404 Not Found
```

Error body:

```json
{
  "detail": "Task not found"
}
```

---

### Update Task

```http
PATCH /api/tasks/{task_id}
```

Request may contain any subset of:

```json
{
  "title": "Updated title",
  "description": "Updated description",
  "status": "in_progress"
}
```

Behavior:

* update only provided fields,
* update `updated_at`,
* if status changes to `done`, set `completed_at`,
* if status changes from `done` to another status, clear `completed_at`.

If the task does not exist, return `404`.

---

### Delete Task

```http
DELETE /api/tasks/{task_id}
```

If the task exists:

```text
204 No Content
```

If the task does not exist:

```text
404 Not Found
```

---

### Mark Task Done

```http
POST /api/tasks/{task_id}/done
```

Behavior:

* set status to `done`,
* set `completed_at`,
* update `updated_at`.

If the task does not exist, return `404`.

---

## 4. Error Handling

Use `HTTPException`.

Use consistent messages:

```text
Task not found
No open task found
```

Do not leak internal errors.

## 5. Tests

Add API tests.

Create:

```text
tests/test_api_tasks.py
```

Use FastAPI `TestClient`.

Tests must use an isolated temporary or in-memory SQLite database.

Required tests:

* `GET /health` returns status ok,
* create task,
* list tasks,
* filter tasks by status,
* get existing task,
* get missing task returns 404,
* update task,
* update missing task returns 404,
* delete task,
* delete missing task returns 404,
* mark task done,
* mark missing task done returns 404,
* get next open task,
* get next open task returns 404 when no open tasks exist.

Existing tests from v0.1.0 must continue to pass.
