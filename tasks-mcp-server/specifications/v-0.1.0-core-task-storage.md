# Specification: v0.1.0 - Core Task Storage

## Goal

Create the initial version of the `tasks-mcp-server` project.

This version focuses only on the backend core:

* project structure,
* SQLite database,
* task domain model,
* repository/service layer,
* basic tests,
* semantic versioning,
* Git workflow.

Do not implement REST API, frontend, or MCP tools in this version.

---

## Version

Target version:

```text
0.1.0
```

```text
tasks-mcp-server
```

Initialize it as a Git repository if needed.

## Functional Requirements

### 1. Task Model

Create a `Task` database model with these fields:

```text
id: integer primary key
title: string, required
description: string, optional
status: string, required
created_at: datetime
updated_at: datetime
completed_at: datetime, optional
```

Supported statuses:

```text
open
in_progress
done
cancelled
```

Use a Python enum for task status.

Default status:

```text
open
```

---

### 2. Database Layer

Use SQLite.

Create database helpers in:

```text
src/tasks_mcp_server/database.py
```

Required functionality:

```text
create_db_and_tables()
get_engine()
get_session()
```

Use SQLModel.

The database file should default to:

```text
tasks.db
```

Allow overriding the database URL through environment variable:

```text
TASKS_MCP_DATABASE_URL
```

Example:

```text
sqlite:///tasks.db
```

Tests must be able to use an in-memory SQLite database.

---

### 3. Schemas

Create simple input/output schemas in:

```text
src/tasks_mcp_server/schemas.py
```

Required schemas:

```text
TaskCreate
TaskUpdate
TaskRead
```

`TaskCreate`:

```text
title: str
description: str | None = None
```

`TaskUpdate`:

```text
title: str | None = None
description: str | None = None
status: TaskStatus | None = None
```

`TaskRead` should represent the public task data.

---

### 4. Repository Layer

Create repository functions or a small repository class in:

```text
src/tasks_mcp_server/repositories.py
```

Required operations:

```text
create_task(session, task_create)
get_task(session, task_id)
list_tasks(session)
update_task(session, task_id, task_update)
delete_task(session, task_id)
get_next_open_task(session)
mark_task_done(session, task_id)
```

Expected behavior:

* `list_tasks` returns all tasks ordered by creation time descending.
* `get_next_open_task` returns the oldest task with status `open`.
* `mark_task_done` sets:

  * `status = done`
  * `completed_at = current UTC datetime`
  * updates `updated_at`

---

### 5. Service Layer

Create a thin service layer in:

```text
src/tasks_mcp_server/services.py
```

The service layer should wrap repository operations and contain business logic.

Required service class:

```text
TaskService
```

Required methods:

```text
create_task(...)
get_task(...)
list_tasks(...)
update_task(...)
delete_task(...)
get_next_open_task(...)
mark_task_done(...)
```

The service should be used by tests.

## Git Workflow

### 1. Initialize Repository

If the directory is not a Git repository, initialize it:

```bash
git init
```

Create or verify the `main` branch.

```bash
git branch -M main
```

If a remote already exists, keep it.

If no remote exists, stop and report that the GitHub remote is missing.

---

### 2. Create Feature Branch

Create a dedicated branch for this version:

```bash
git checkout -b feature/v0.1.0-core-task-storage
```

If the branch already exists, switch to it.

---

### 3. Implement Version

Implement all requirements from this specification.

---

### 4. Verify

Before committing, run:

```bash
ruff check .
pytest
```

Both must pass.

If formatting issues are found and ruff is configured to fix them safely, run:

```bash
ruff check . --fix
```

Then rerun:

```bash
ruff check .
pytest
```

---

### 5. Commit

Create a commit after successful verification:

```bash
git add .
git commit -m "feat: implement core task storage v0.1.0"
```

---

### 6. Push

Push the branch to GitHub:

```bash
git push -u origin feature/v0.1.0-core-task-storage
```

---

### 7. Merge Rule

Do not merge automatically.

After pushing, report the branch name and wait for human approval.

Only after explicit human approval, merge into `main`.

If the human requests patches, implement all requested patches on the same feature branch.

The merge to `main` should happen only after all patches for version `0.1.0` are completed and approved.
